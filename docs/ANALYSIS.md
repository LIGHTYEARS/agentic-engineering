# 讨论记录核心分析

> 基于 `docs/references/` 中 21 篇文档的系统性分析

---

## 一、总论

这份讨论记录构建了一条完整的从**数学本质 → 认识论基础 → 工程架构**的论证链，最终产出一套面向单团队落地的「九层人-AI 协作交付体系」。

**一句话概括**：AI Coding Agent 面临的根本挑战是**多目标非凸优化问题**——搜索空间充满局部最优（非凸），且多个目标互相冲突（多目标）；而 LLM 的自回归范式天然不适合解决此类问题。解法不是修模型，而是在模型外部构建一个可验证、可回滚、可并行搜索的**复合工程系统**。

---

## 二、论证的四幕结构

### 第一幕：数学基础与机制缺陷（文档 01–04）

**核心命题**：真实编码任务（如「修复并发 Bug 且不损害性能/兼容性/资源」）本质上是多目标非凸优化。LLM 的自回归生成——逐 token 局部决策、单轨迹展开、无显式目标函数——天然不适合此类问题。

**关键洞察**：
1. 形式化定义：5+ 个互斥约束在非凸空间中，不存在唯一最优解，只有帕累托前沿上的妥协解集
2. 自回归的五大结构性缺陷：目标仅存于自然语言、每步为局部决策、路径依赖锁定、单轨迹展开、缺乏外部目标函数
3. **核心转折**：解法不是「修复自回归」，而是保留 LLM 作为方案生成器，在外部搭建 compound AI system

### 第二幕：工程挑战与策略（文档 05–06）

**核心命题**：Coding agent 从 Demo 到生产的瓶颈已非代码生成能力，而是缺少一个完整的工程控制系统来约束、观察、验证、协调、回滚并持续改进 agent。

**关键洞察**：
1. **硬约束不可补偿**：正确性/安全性/兼容性是淘汰条件，不是加权维度
2. **代码级融合极少可行**：两个分支若基于互斥架构假设，融合是「未验证的新方案」，必须从头评测
3. **开放式评测六层模型**：能力契约 → 不变量测试 → 场景矩阵 → 变形测试 → 差分评测 → rubric 主观质量
4. **实现者不应设计自己的测试**：独立 Test Designer Agent + Adversarial Agent 是结构性防线
5. **P0 问题**：早期错误检测（引入「认知单元测试」）和权限爆炸半径控制
6. **瓶颈转移**：agent 的速度只是把「开发瓶颈」变成「评审/集成/发布瓶颈」

### 第三幕：哲学桥梁——语义与事实的鸿沟（文档 07–10）

**核心命题**：语义连贯性 ≠ 事实真值。LLM 训练目标是最大化 P(token|上下文)，而非判定 P(命题为真|世界状态)。这一鸿沟是结构性的，不会因模型规模增大而消失。

**关键洞察**：
1. **四层不等式**：语法正确 ≠ 语义完整 ≠ 命题为真 ≠ 有证据确认为真
2. **多对一有损映射**：现实→语义的编码路径 R→O→D→A→S 是多对一的，语义不可逆推现实
3. **任务充分性替代完整等价**：工程上追求「S 对任务 T 足够」，而非 S≡R
4. **证据齐全不保证推导正确**：LLM(E)→Ĉ 没有 soundness 保证，chain-of-thought 可能是事后合理化
5. **确定性外移原则**：能确定性计算的→程序；能规则判定的→policy engine；开放式综合→LLM+监督
6. **proof-carrying claim**：Agent 输出应是「结论+前提+证据+规则+范围」的结构体，可被独立校验

**这一幕的核心作用**：为第四幕的架构设计提供认识论基础——解释了*为什么*需要将验证确定性化、为什么 LLM 角色必须收缩为「不完全可靠的假设生成器」。

### 第四幕：九层协作框架（文档 11–21）

**核心命题**：将多目标非凸优化问题分解为 9 个具有清晰输入输出契约的平面，通过职责分离、确定性验证与概率生成分离、人类价值判断与 AI 执行能力分离，构建可治理的离散决策链。

**九层结构**：

| # | Plane | 核心职责 | 解决的认识论问题 |
|---|-------|----------|-----------------|
| 1 | Governance | 权限边界、风险分级、审批矩阵 | 「能做什么」的确定性约束 |
| 2 | Spec | 需求拆解、事实/假设分离、硬约束 vs 软目标 | 将隐性目标外化为可验证契约 |
| 3 | Reality | 证据采集、7种认知状态分类、provenance | 「现在是什么」的事实建立 |
| 4 | Orchestration | 任务 DAG、agent 分配、Context Pack | 搜索组织与资源分配 |
| 5 | Search | 候选假设驱动、多样性控制、终止条件 | 并行探索减少路径依赖 |
| 6 | Evaluation | 6层验证金字塔、差分验证、轨迹评测 | 将「测出来」确定性化 |
| 7 | Decision | 帕累托非支配比较、偏好显式化、ADR | 在不确定性中做可审计选择 |
| 8 | Delivery | 渐进暴露阶梯、运行时信号、回滚 | 将选择安全地映射到生产 |
| 9 | Learning | 失败分类(12类)、资产版本化、eval 扩展 | 闭环积累与知识淘汰 |

**关键设计原则**：
1. 隔离内自治、边界处审批（安全与效率的帕累托最优点）
2. 搜索/评测/决策三分离（防止自我评价偏差）
3. 决策是时间绑定的组织承诺，不是永久事实（可重开）
4. Learning Plane 有「遗忘」能力（防规则无限膨胀）
5. 仓库即治理——所有产物版本化存于 Git，利用现有基础设施

## 三、贯穿全文的核心观点

### 观点 1：不修模型，修系统

> 自回归不是 bug，是计算范式的边界条件。解法是改变系统架构，不是改变模型。

LLM 保持其角色为「高能力但不完全可靠的假设生成器」，所有可靠性保证由外部工程系统提供。

### 观点 2：瓶颈已从生成转移到验证

> 写出五个候选不难，判断哪个好才难。

Agent 的上限取决于仓库的 verifier 完备度，而非模型的生成能力。同一模型在强约束环境 vs 弱测试环境中表现判若两人。

### 观点 3：硬约束先于软目标

> 不是给十个目标加权求和，而是先做 constraint satisfaction（不满足则淘汰），再在可行解中比较软目标。

这是多目标优化在工程中的最重要落地方法论。

### 观点 4：确定性能力必须下沉

> 越关键的约束越应下沉到确定性机制：编译器 > 类型系统 > 测试 > LLM reviewer > 人类。不能用 LLM 判断 LLM。

### 观点 5：语义连贯 ≠ 事实正确——这是结构性的，不是规模能解决的

> LLM 学的是 P(token|文本上下文)，不是 P(命题为真|世界状态)。一致性只证明分布倾向稳定，不证明命题为真。

### 观点 6：Agent 的真实价值指标是「可信变更的单位成本」

> 净收益 = 节省工程时间 − 评审成本 − 返工成本 − 运行成本 − 事故预期损失

而非生成代码行数或 SWE-bench 分数。

## 四、创新之处

1. **跨学科类比的结构性同构**：将多目标优化（帕累托前沿、非支配排序）严格映射到软件工程决策，而非泛泛比喻
2. **认识论方法论系统化嵌入工程**：七种认知状态、证据五属性、四层不等式——将科学方法论转化为可操作约束
3. **proof-carrying claim 概念**：将 proof-carrying code 思想迁移到 AI agent 输出设计
4. **Goodhart 定律的工程化防御**：隐藏测试 + 独立设计 + Adversarial Agent + 轨迹评测的多层防线
5. **「仓库即治理」的务实哲学**：拒绝「先做平台」，利用 Git/CI/PR 现有基础设施
6. **Learning Plane 的遗忘能力**：资产淘汰机制防止规则膨胀导致上下文污染
7. **认知单元测试**：限制无验证连续操作步数——agent 行为控制的具体工程机制

## 五、局限与未解决问题

1. **验证成本可行性**：完整九层体系的协调开销在小规模需求上可能过重，缺少判断何时简化的决策框架
2. **人类瓶颈未量化**：多个关键节点要求人类审批，但未分析单人可支撑的吞吐量
3. **候选多样性**：同模型同提示的并行 agent 可能生成同质化方案，有效多样性策略未深入
4. **长期目标不可测**：运维复杂度、内存碎片、团队维护成本需部署后长期观测，PR 阶段无法验证
5. **形式化停留在类比**：虽用数学公式，但未建立可计算的目标函数或可证明的收敛保证
6. **Gate 间的阻塞**：框架倾向线性瀑布推进，跨 Plane 快速迭代的具体机制不足
7. **跨团队/跨服务未覆盖**：明确限定单团队，微服务环境下的协商边界未讨论
8. **缺乏实证数据**：论证依赖哲学框架和工程直觉，无具体案例的量化改善证据

## 六、总结：这份讨论的知识贡献

这份讨论记录的核心价值在于它构建了一条从**数学问题定义**经**认识论分析**到**可落地工程架构**的完整推导链：

```
多目标非凸优化（问题定义）
        ↓
自回归的结构性缺陷（根因分析）
        ↓
语义≠事实的不可消除鸿沟（认识论基础）
        ↓
确定性外移 + 候选并行 + verifier 驱动（设计原则）
        ↓
九层协作框架（架构方案）
        ↓
仓库即治理、渐进式暴露、闭环学习（落地路径）
```

它不是一个关于「AI 能不能写代码」的讨论，而是一个关于**「如何在 LLM 能力边界已知的前提下，设计一个可治理、可验证、可持续改进的人-AI 协作工程系统」**的系统性思考。
