# Coding Agent 工作流中的语义鸿沟

> Source: 多目标非凸优化解析讨论记录
> Date: 2026/7/15 19:29:19

---

## Prompt

这反映在大型代码仓库中工作的 ai coding agents 的工作流程设计与编排上表现是怎样的，假如现在来了一个存量模块增加新功能的需求。

---

## Response

这类问题在大型代码仓库里，最终会把 coding agent 的工作流从“读需求→改代码”改造成：

需求语义 → 现实约束采集 → 可执行规格 → 候选实现 → 证据验证 → 灰度校正

因为“给存量模块增加新功能”这句话本身只是语义描述，它并不自动等价于：

真实用户需要什么；

现有模块实际上怎么运行；

哪些行为不能破坏；

哪些隐性依赖存在；

什么叫“实现完成”；

上线后是否真的有效。

所以工程编排的核心，是逐层缩小“需求文本”和“真实系统状态”之间的差距。

一、错误的简单流程

很多 Agent 默认会这样工作：

需求文本
→ 搜索相关文件
→ 推断实现方式
→ 修改代码
→ 跑现有测试
→ 提交 PR

问题在于，它默认做了很多未经验证的等价转换：

“用户需要导出功能”
≈
“加一个导出按钮”

“兼容现有模块”
≈
“现有测试通过”

“性能不能下降”
≈
“代码看起来复杂度不高”

“符合业务流程”
≈
“与附近代码风格一致”

这些只是语义上合理的推测，不一定对应真实系统。

二、正确流程应增加一个“现实建模阶段”

假设需求是：

给存量订单模块增加“批量取消订单”功能。

Agent 不应该马上实现，而应先建立一个任务现实模型。

1. 识别需求中的语义主张

先拆出需求声称了什么：

feature:
  name: 批量取消订单

claimed_capabilities:
  - 用户可以选择多个订单
  - 用户可以一次提交取消
  - 系统返回处理结果

这只是语言层。

2. 识别哪些事实尚未确认

Agent 应主动列出未知项：

unknowns:
  - 哪些订单状态允许取消
  - 是否要求全部成功或允许部分成功
  - 最大批量数量
  - 是否需要幂等
  - 是否有退款流程
  - 是否触发库存回滚
  - 是否发送通知
  - 是否涉及权限差异
  - 是否已有下游依赖单笔取消事件

这一步很关键。

一个成熟 Agent 不应把“没写”理解成“不存在”。

三、先做仓库现实侦察，而不是代码搜索

大型仓库里的真实语义，通常分散在很多地方：

代码；

测试；

数据库 schema；

API；

配置；

事件定义；

监控；
-历史 PR；

文档；

线上行为；

团队约定。

因此 Agent 的探索流程应是多源的。

代码层

寻找：

现有单笔取消逻辑；

订单状态机；

权限检查；

事务边界；

退款与库存调用；

事件发布；

审计日志。

测试层

寻找：

哪些行为被现有测试保护；

哪些边界没有覆盖；

是否存在隐含业务规则；

是否有 snapshot 或 contract test。

运行层

获取：

当前接口调用量；

单笔取消延迟；

失败率；

热点租户；

下游超时情况；

当前批处理任务模式。

历史层

查看：

过去为什么没有批量接口；

是否曾出现过相关事故；

是否有被拒绝的类似方案；

哪些模块最容易回归。

这一步的产物不应该只是“我看了这些文件”，而应该是一份结构化事实清单。

observed_facts:
  - single_cancel_is_transactional: true
  - inventory_rollback_is_async: true
  - refund_is_not_idempotent: false
  - max_downstream_qps: 200
  - cancellation_event_has_single_order_schema: true

evidence:
  - file: order_cancel_service.ts
  - test: cancel_order.integration.spec.ts
  - metric: inventory-service-rate-limit
  - incident: INC-2025-184

四、把需求变成“可执行规格”

这一步是整个编排的核心。

Agent 需要把自然语言需求转换为三类东西：

1. 能力规格

capabilities:
  - 支持一次提交多个订单
  - 返回每个订单的处理结果
  - 允许部分成功
  - 失败项可以重试

2. 硬约束

hard_constraints:
  - 不允许绕过单笔取消的业务校验
  - 不允许重复退款
  - 不允许重复库存回滚
  - 现有单笔接口行为不变
  - 旧事件消费者必须继续工作
  - 批量上限 <= 100

3. 可测量目标

measurable_targets:
  - 100 个订单请求 P95 < 2s
  - 下游库存服务 QPS 不超过 180
  - 重试不能产生重复副作用
  - 现有集成测试全部通过

这时需求才从“语义”变成一组能验证的工程命题。

五、Agent 编排要做职责分离

不应该让同一个 Agent 同时：

理解需求；

设计方案；

编写实现；

编写测试；

判断自己是否正确。

更合理的编排是：

Requirement Agent
→ Repository Analyst
→ Test Designer
→ Implementation Agents
→ Verifier
→ Reviewer
→ Human

Requirement Agent

负责：

拆解能力；

识别歧义；

标记未知；

确定验收边界。

Repository Analyst

负责：

获取仓库事实；

建依赖图；

找隐含约束；

输出证据。

Test Designer

在看不到候选实现细节前设计：

黑盒测试；

不变量；

边界场景；

回归矩阵；

性能 baseline。

Implementation Agents

分别探索不同实现：

同步批量；

并行受限执行；

异步任务模式；

现有单笔接口编排；

新建批处理领域服务。

Verifier

统一执行：

功能测试；

幂等测试；

并发测试；

契约测试；

benchmark；

下游流量模拟。

Reviewer

关注：

是否引入新架构概念；

是否破坏领域边界；

是否存在无法测试的假设；

是否具备回滚能力。

六、必须建立 baseline，否则“提升”和“无回归”没有意义

对于存量模块，新功能最容易忽略的不是新增行为，而是原有行为。

在改代码前，应记录：

baseline:
  single_cancel_p95: 120ms
  error_rate: 0.08%
  inventory_qps_peak: 140
  refund_duplicate_rate: 0
  event_delivery_success: 99.98%

还要保存当前行为样本：

合法状态取消
非法状态取消
重复取消
退款失败
库存回滚超时
事件发布失败
权限不足

这样改造后才能做差分。

旧系统行为
vs
新系统单笔行为
vs
新系统批量行为

没有 baseline，Agent 很容易把“功能增加了”误判成“系统变好了”。

七、开放式需求要用“不变量”而不是固定实现评测

批量取消可以有很多正确实现，不能写死：

必须使用队列
必须新增 BatchCancelService
必须返回某个固定 JSON 结构

应该评测：

所有合法订单最终都进入已取消状态

任何订单最多退款一次

任何订单最多回滚库存一次

失败订单不会影响已成功订单

相同请求重试不会产生额外副作用

单笔取消行为保持不变

这些是不变量。

它们比“必须怎么写”更接近真实需求。

工程流程应该：

先淘汰违反硬约束的方案；

比较剩余方案的帕累托关系；

判断优势是否能组合；

根据业务偏好决定；

必要时保留多模式。

例如：

<= 10 单
→ 同步有界并发

> 10 单
→ 异步任务模式

这时最优解不是“一个实现”，而是运行时策略。

九、语义和事实之间的差距，应通过“证据链”管理

一个成熟 Agent 的结论不能只是：

“实现完成，测试通过，性能良好。”

而应该输出：

claim:
  "批量取消支持 100 单，P95 小于 2 秒"

evidence:
  benchmark_run: batch-cancel-2026-07-15-03
  environment: staging-8c16g
  sample_size: 30
  p95: 1.82s

limitations:
  - 未验证生产热点租户流量
  - 未覆盖库存服务区域故障

也就是说，Agent 的输出要从“自然语言结论”升级为：

主张
+
证据
+
适用范围
+
不确定性
+
限制

十、上线不是结束，而是继续校准现实

有些事实无法在 PR 阶段确认：

生产流量分布；

真实用户操作模式；

下游服务竞争；

长尾延迟；

灰度期间的数据一致性；

用户是否真的会使用该功能。

因此编排要继续到运行时：

Draft PR
→ Staging
→ Shadow
→ 1% Canary
→ 10% Canary
→ Full Rollout

每一阶段都有停止条件：

rollback_if:
  duplicate_refund > 0
  batch_cancel_error_rate > 1%
  inventory_qps > 180
  single_cancel_p95_regression > 5%

线上数据还要回灌下一轮评测集。

十一、完整流程可以表示为

需求文本
  ↓
语义拆解
  ↓
未知事实识别
  ↓
仓库与运行现实采集
  ↓
结构化规格
  ↓
独立评测设计
  ↓
并行候选实现
  ↓
统一证据验证
  ↓
方案选择或融合
  ↓
PR 与人工决策
  ↓
灰度与生产观测
  ↓
事实回灌与规格更新

十二、这对 Agent 编排提出的核心原则

1. 不把需求文本当事实

需求是主张，不是系统状态。

2. 不把代码附近的模式当业务真相

相似实现只是先验，不是证据。

3. 不把测试通过当需求完成

测试覆盖的是已编码的规格，不是全部现实。

4. 不把来源忠实当来源真实

文档、注释、历史 PR 都可能过期。

5. 不把 Agent 的解释当测量结果

性能、安全、兼容性必须由工具验证。

6. 不把一次上线当长期正确

需要运行时监控、反馈和修正。

最终抽象

大型代码仓库中，一个 coding agent 的工作不应被定义为：

根据需求文本生成代码。

而应该被定义为：

根据需求主张，主动采集仓库与运行环境中的相关事实，把它们转化为可执行规格和评测，再通过候选搜索、证据验证与生产反馈，逐步缩小语义描述与真实系统行为之间的差距。

因此，一个成熟的 Agent 系统，本质上不是“代码生成器”，而是一个：

$$
\boxed{
\text{语义解释器}
+
\text{事实采集器}
+
\text{方案搜索器}
+
\text{证据验证器}
+
\text{运行时校准器}
}
$$

这正是“语义无法完整等价现实”在工程流程设计上的直接体现。
