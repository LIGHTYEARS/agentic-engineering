你的直觉基本正确，但建议把“语义”和“事实”之间再拆一层。更准确的表述不是“语义事实与客观事实不等价”，而是：

一个句子具有完整、合理的语义，不代表它所表达的命题为真。命题是否为真，还取决于它是否与特定世界状态相符合，以及我们是否拥有足够证据确认这种符合关系。

这对应哲学上的真理符合论：一个命题为真，不是因为它听起来合理或与其他句子协调，而是因为它与现实中的事实相对应。(斯坦福哲学百科全书)

一、需要区分四个层次

以今天的天气为例。

1. 句法层

今天是个大晴天。
今天是个下雨天。

两句话都符合汉语语法。

2. 语义层

两句话也都有清晰的意义：

“今天”指向一个时间；

“天气”隐含一个地点；

“晴天”和“雨天”描述天气状态。

把“晴天”换成“雨天”，不会破坏句子的语义结构。

3. 命题层

它们表达两个不同命题：

$$
p = \text{今天这里是晴天}
$$

$$
q = \text{今天这里在下雨}
$$

如果时间和地点固定，这两个命题通常不能同时为真。

4. 事实与证据层

要判断 $p$ 或 $q$，必须接触句子之外的信息：

看窗外；

查询气象站；

获取卫星或雷达数据；

确认时间和地点；

判断“正在下雨”的具体标准。

因此：

语法正确
≠ 语义完整
≠ 命题为真
≠ 我们有证据知道它为真

这四件事经常被混在一起。

二、你的“30% 换成 50%”是一个很典型的例子

考虑：

改造后性能提升了 30%。
改造后性能提升了 50%。

二者：

句法都正确；

语义都完整；

工程语境里都很自然；

都可能出现在真实技术报告中；

都符合“优化带来性能提升”这种常见叙事模式。

但它们表达的是两个互斥的数量命题。

要判断哪个是真的，需要补齐测量定义：

metric: throughput
baseline: 10,000 req/s
candidate: 13,000 req/s
environment:
  cpu: 8 cores
  concurrency: 200
  dataset: production-sample-v4
aggregation: median of 10 runs
warmup: 120s

然后才能计算：

$$
\frac{13000-10000}{10000}=30%
$$

“提升了 50%”在语言空间中完全合理，但不被上述数据支持。

可以把这个差别概括为：

LLM 很擅长判断一个数字放在这里像不像真的，但“像真的”不是“由测量推出的”。

研究也确实将语言模型的概率与 semantic plausibility，即语义合理性或事件合理性联系起来。模型可以根据文本分布判断“演员获得奖项”比“演员获得战役”更合理，但这类合理性只是世界知识的代理信号，不等同于对当前具体事实的验证。(arXiv)

三、这涉及三个相关但不同的问题

1. Form 与 meaning 的区别

Bender 和 Koller 在经典论文中强调，模型从文本中直接获得的是语言形式之间的分布关系；仅仅从形式训练，不能自动推出系统拥有与现实交互所产生的完整意义。(ACL Anthology)

例如训练数据可能包含：

性能提升了 10%
性能提升了 30%
性能提升了 50%
性能提升了 300%

模型可以学习到：

“性能提升了 [数字]%”

是一个常见且语义自然的模式。

但训练目标通常没有要求模型在生成数字前，重新运行对应 benchmark。

所以模型学到的是：

$$
P(\text{token} \mid \text{文本上下文})
$$

而不是：

$$
P(\text{命题为真} \mid \text{现实世界状态})
$$

这两个概率可能相关，但绝不相同。

2. Symbol grounding：符号如何连接现实

Harnad 提出的“符号落地问题”问的是：

如果一个系统中的符号只通过其他符号获得解释，那么它最终怎样连接到符号之外的对象和现实？

他用类似“只靠一本中文-中文词典学中文”的思路说明：每个词都由更多词解释，可能始终没有接触词语所指向的对象。(MIT Media Courses)

对应到 LLM：

“雨”
→ 被其他文本中的“云、湿、降水、伞”解释

但“现在北京是否正在下雨”要求的是：

“雨”这个概念
+
北京
+
当前时间
+
真实气象观测

所以问题不只是“模型懂不懂雨是什么意思”，还包括：

它当前生成的这个“雨”，是否正确指向此时此地的现实状态？

这通常被称作：

referential grounding：指称落地；

sensorimotor grounding：感知-行动落地；

epistemic grounding：认知或证据落地。

相关研究特别提醒，不同“grounding”概念经常被混用。(arXiv)

3. Truthfulness：训练分布中的话不一定是真的

训练文本不仅包含真实信息，也包含：

误解；

传闻；

过时信息；

广告夸大；

错误数字；

虚构文本；

复制传播的常见谬误。

TruthfulQA 的设计目标正是测试模型是否会重复人类文本中的常见错误观念。论文发现，模型会生成从训练分布中模仿来的错误答案，说明更擅长拟合语言分布并不自动等于更真实。(ACL Anthology)

因此模型可能同时学到：

“喝酒可以让身体暖和”

是一句常见且流畅的话，以及与它相关的大量叙述。

但“常见”是文本分布属性，不是生理事实的证明。

四、“语义事实”这个词不够精确

你提出的“语义事实”和“客观事实”方向上能表达意思，但在哲学和语义学中，更推荐使用下面这组词。

你想表达的东西
更准确的称呼
句子表达了什么
命题内容
句子在什么条件下为真
真值条件
现实中实际发生了什么
世界状态 / 事态
传感器或工具返回了什么
观测 / 测量记录
文本声称发生了什么
陈述 / 主张
主张有数据、来源支持
有依据的主张
主张与现实相符合
真实命题

可以画成：

现实世界状态
      ↓ 观测
传感器 / 数据库 / 实验
      ↓ 编码
数据记录
      ↓ 解释
命题或结论
      ↓ 表达
自然语言文本

例如：

现实状态：
服务器在特定负载下处理 13,000 req/s

测量记录：
benchmark run #142 = 13,000 req/s

计算结论：
相对 10,000 req/s 提升 30%

语言陈述：
“改造后性能提升了 30%”

最下面的文本不是事实本身，而是对事实链条的语言表达。

五、你说“需要将事实补充为其他语义形式”，这个判断也很关键

LLM 不能直接把物理世界放进上下文。

它只能接收被编码后的表示：

天气传感器
→ JSON
→ token

性能压测
→ 日志和表格
→ token

数据库状态
→ 查询结果
→ token

代码运行
→ exit code / stdout
→ token

因此工具使用或 RAG 做的事情可以描述为：

把外部世界的一部分状态，转换为模型可以处理的符号证据。

例如：

{
  "baseline_qps": 10000,
  "candidate_qps": 13000,
  "runs": 10,
  "confidence_interval": [28.7, 31.5]
}

这比只告诉模型“性能有所提升”可靠得多。

但是你说得对，即使外部事实被转换成文本，也不能自动保证结论真实，因为中间还有很多潜在错误：

传感器可能有误；

查询可能查错环境；

benchmark 可能设计不合理；

数据可能过期；

JSON 可能被截断；

单位可能被误解；

模型可能读错结果；

来源文本本身可能造假；

结论可能超过数据支持范围。

所以 grounding 并没有消除真理问题，只是把问题从：

模型纯猜

改善为：

模型依据某些外部证据作判断

Google DeepMind 的 FACTS Grounding 评测也采用类似区分：它评估模型能否忠实依据提供的来源回答并避免无来源陈述，但“忠实于来源”仍不意味着来源本身就是现实世界的最终真相。(Google DeepMind)

六、证据文本仍然不是事实本身

这是你论述里最深的一层。

假设上下文中提供：

测试报告：性能提升 50%。

模型可以做到“忠实引用”：

根据测试报告，性能提升了 50%。

这句话可能忠实于文档。

但文档可能：

测错了；

使用有偏 workload；

只挑最好的一次；

基线配置不同；

把 latency 下降误写成 throughput 上升；

是人工伪造的。

于是存在三个不同问题：

来源忠实性

模型有没有正确复述来源？

来源可靠性

来源本身是否值得相信？

现实符合性

来源的内容是否真的对应现实？

它们不能相互替代：

$$
\text{忠实于来源}
\nRightarrow
\text{来源可靠}
\nRightarrow
\text{命题真实}
$$

更准确地说，这两条蕴含都不成立。

七、代码为什么看起来与普通事实不一样？

你说代码有时“语义可能和事实等价，因为代码正确性来源于文本本身”，这里需要稍微修正。

代码确实有一类“内部可判定事实”

例如：

function add(a, b) {
  return a + b;
}

在给定 JavaScript 规范和运行时语义下，它的行为可以由程序文本和形式规则推导。

这类事实接近：

这个表达式会返回什么？
这个类型是否成立？
这个控制流是否可达？
这个程序是否通过某个测试？

它们的判定环境相对封闭：

$$
\text{程序文本}
+
\text{语言语义}
+
\text{输入}
\rightarrow
\text{执行结果}
$$

程序文本具有操作语义：代码不是只在描述一个可能世界，它还可以在解释器或机器上被执行。

因此对某些问题：

代码
→ 运行
→ 结果

形成了非常短的验证闭环。

但“代码正确性来源于文本本身”仍然只对一部分成立

程序是否“正确”仍需要外部规格。

例如：

function calculateDiscount(price) {
  return price * 0.8;
}

代码非常明确地实现了八折。

但它是否正确取决于现实业务规则究竟是：

八折；

满 100 八折；

VIP 八折；

某地区禁止折扣；

税前八折还是税后八折。

代码文本决定它做了什么，却不能单独决定它做的是否是业务想要的事。

所以应该区分：

问题
依据
代码有没有语法错误
语言语法
代码执行什么行为
操作语义与运行时
代码是否满足形式规格
程序 + 规格
代码是否满足真实业务需求
程序 + 规格 + 现实业务
代码在线上是否性能达标
真实 workload 与测量
代码是否长期可靠
生产环境与时间

代码只是在某些封闭问题上拥有非常强的可执行 oracle，而不是语义和事实天然完全等价。

九、为什么“多生成几次并自洽”也不能解决这个问题？

假设询问当前天气，模型十次都回答：

今天是晴天。

这种一致性最多说明：

模型有稳定倾向；

某个答案在语言分布中概率高；

模型内部不太犹豫。

它并不能证明窗外真的晴。

语义熵研究能够检测模型对同一问题产生多个不同意义答案时的“confabulation”，但论文明确只声称检测一部分幻觉；一个模型也可能非常一致地给出同一个错误答案。(Nature)

因此：

$$
\text{一致性}
\neq
\text{真实性}
$$

$$
\text{置信度}
\neq
\text{证据强度}
$$

$$
\text{语义合理性}
\neq
\text{现实符合性}
$$

十、如何把你的观点整理成一套较严谨的表述

可以改写为：

大语言模型主要从文本分布中学习语言形式、概念关系和命题的语义合理性。一个陈述在语法和语义上成立，只表示它是一个可理解、上下文连贯且可能发生的命题，并不表示它与当前现实状态相符合。

对于封闭、形式化、可执行的问题，例如给定输入下的代码行为，系统可以通过解释器、类型系统或测试建立较短的语义-执行验证闭环。但对于天气、性能、库存、用户行为等开放世界事实，真实性依赖外部观测、测量和来源。

工具可以把现实世界的一部分状态编码为文本或结构化数据，再交给模型分析；但这一过程只能提供有限且可能有误的证据表示，而不是把现实本身放入上下文。因此，模型输出的可靠性取决于观测质量、来源可信度、测量设计、数据时效性和推理过程，而不仅是语言模型的语义能力。

我认为这比“语义事实和客观事实不等价”更精确。

十一、一个适合工程实践的五层模型

以后评估 Agent 的一句结论，可以逐层追问：

第 1 层：这个句子语义通顺吗？
第 2 层：它表达的命题是什么？
第 3 层：这个命题需要哪些现实条件才为真？
第 4 层：有哪些观测或证据支持这些条件？
第 5 层：证据来源和推导过程可靠吗？

以“性能提升了 30%”为例：

语义：
改造带来性能改善

命题：
候选系统相对基线提升 30%

真值条件：
同一指标、同一环境、同一 workload 下，
(candidate - baseline) / baseline = 30%

证据：
benchmark 原始结果、环境配置、多次运行统计

可靠性：
测试是否公平、结果是否显著、是否可复现、
是否存在 cherry-pick、是否符合生产流量

如果系统只能回答第一层，它生成的是合理文本。

如果能回答前两层，它在处理命题。

如果能完成第三至第五层，它才开始形成有现实依据的知识判断。

最终可以压缩成一句话：

语言模型的默认能力是生成语义上可能成立的命题；工程系统的责任，是通过观测、来源、执行和验证，把“可能成立”收缩为“有证据支持”，并明确仍然无法确认的部分。
