# 九层协作框架总览

> Source: 多目标非凸优化解析讨论记录
> Date: 2026/7/15 20:07:20

---

## Prompt

根据前面的讨论和研究，那么在工程上，我要在团队内落地这么一套人与 AI 协作半自动化需求交付体系，可以划分为多少个 plane，每个 plane 的具体职责是什么，以表格给出回复？

---

## Response

建议划分为 9 个 Plane。其中前 8 个覆盖一次需求从进入到上线的完整生命周期，第 9 个负责把每次交付结果沉淀为团队长期能力。

1. Governance Plane
治理与风险平面
定义 AI 能做什么、不能做什么；配置权限、数据边界、工具白名单、审批等级、成本预算和责任归属
权限策略、风险等级、操作 Guardrail、审批矩阵、预算规则、审计规范
人制定风险政策并承担最终责任；AI 必须在授权范围内工作，不能自行扩大权限
2. Spec Plane
需求与规格平面
将自然语言需求转化为能力、约束、不变量、验收标准、优先级和明确的未知项；区分需求主张与已确认事实
Feature Spec、Acceptance Criteria、硬约束、软目标、不变量、Open Questions、需求版本
AI 辅助拆解需求、发现歧义；产品、业务和技术负责人确认需求语义与价值判断
3. Reality Plane
事实与上下文平面
从代码库、文档、测试、历史 PR、监控、数据库、事故记录和人员知识中采集现实证据；区分观察事实、推断和假设
Repository Map、依赖图、现状基线、证据清单、假设列表、未知风险、Provenance
AI 负责广泛检索和结构化；领域专家确认隐性规则、历史原因和组织性知识
4. Orchestration Plane
规划与编排平面
将任务拆成有依赖关系的工作单元；选择单 Agent、多 Agent、串行或并行；分配上下文、预算、工具和停止条件
Task DAG、执行计划、角色分工、阶段 Checkpoint、上下文包、预算和超时策略
AI 生成与动态调整计划；Tech Lead 审核高风险路径和架构性任务拆分
5. Search Plane
方案搜索平面
生成差异化候选方案；使用 branch、worktree 或 sandbox 隔离实现；控制候选数量和探索多样性；必要时生成融合候选
设计候选、独立分支、技术取舍说明、Rejected Alternatives、候选版本记录
AI 负责方案探索和代码实现；人定义搜索方向、候选多样性和架构禁区
6. Evaluation Plane
评测与验证平面
构建和执行确定性测试、不变量测试、场景矩阵、变形测试、兼容性检查、benchmark、安全扫描和轨迹评测
Test Suite、Baseline、Benchmark、API Diff、性能与资源报告、Trace、证据包
确定性判断由工具完成；AI 辅助设计测试、解释结果、寻找反例；关键评测不能由实现 Agent 单独控制
7. Decision Plane
选择与评审平面
先按硬约束淘汰不可行方案，再做非支配比较、可组合性分析、业务偏好和风险判断；决定选择、融合、保留多方案或重新探索
候选评估矩阵、帕累托集合、推荐结论、决策依据、ADR、人工审批记录
AI 提供证据化推荐；人负责价值取舍、风险接受、架构决策和最终批准
8. Delivery & Runtime Plane
交付与运行平面
管理 PR、CI、Staging、Shadow、Canary、Feature Flag、正式发布、监控、自动回滚和后续任务
Draft PR、发布证据、灰度计划、SLO、告警规则、回滚策略、生产观测报告
AI 可自动执行低风险发布步骤和监控分析；人批准高风险上线并拥有中止权
9. Learning Plane
学习与资产平面
将成功经验、失败轨迹、事故、人工修改和线上反馈回灌到规则、评测集、Skills、项目文档和编排策略中
AGENTS.md、Skills、Eval Corpus、失败案例库、最佳实践、策略版本、模型与 Prompt 回归报告
AI 负责归纳和生成改进候选；团队审核后将其升级为正式组织资产

Decision

Delivery & Runtime

Learning
这次交付如何让下一次更可靠、更便宜、更快速？

三条关键边界

第一，事实采集和结论推导不能全部交给 LLM。指标计算、约束判断、兼容性检测和测试结果应由确定性工具完成。

第二，实现 Agent 不应同时拥有完整的评测控制权。测试设计、隐藏测试和最终 verifier 至少要保持部分独立。

第三，AI 可以推荐，不能默认承担最终价值判断和风险责任。业务优先级、重大架构变更、生产风险接受仍然需要明确的人类负责人。

这套体系的核心不是建立九个独立系统，而是为每类责任建立清晰边界、输入输出协议和可审计状态。初期可以在一个平台中实现多个 Plane，成熟后再按团队规模拆分。
