# 单向自回归生成的本质与局限

> Source: 多目标非凸优化解析讨论记录
> Date: 2026/7/14 16:00:03

---

## Prompt

这句话怎么理解：“因为 LLM 是单向自回归生成的文本序列，它在输出第 100 行代码的时候，很难像使用进化算法（如 NSGA-II）或粒子群算法那样，同时在脑海里维护并评估 10 个互斥目标的权重方案。”

我不懂的地方在于：什么是单向自回归，以及单向自回归为什么会存在这种不足。

---

## Response

可以把这句话拆成两层：

1. 什么是“单向自回归”

LLM 生成文本时，大致遵循这个过程：

$$
P(x_1,x_2,\dots,x_n)

\prod_{t=1}^{n} P(x_t \mid x_1,\dots,x_{t-1})
$$

意思是：

第 $t$ 个 token，是根据前面已经出现的 token，预测下一个 token。

例如要生成代码：

function add(a, b) {
  return

模型此时会根据前面的内容，预测下一个 token 可能是 a，然后再预测 +，再预测 b。

生成顺序类似：

function → add → ( → a → , → b → ) → { → return → a → + → b

“自回归”是什么意思

“回归”这里可以理解为“预测”；“自”表示它会把自己之前生成的内容，再作为下一步预测的输入。

也就是：

生成第 1 个 token
↓
把第 1 个 token 放回上下文
↓
生成第 2 个 token
↓
把前两个 token 放回上下文
↓
生成第 3 个 token

“单向”是什么意思

生成第 100 行代码时，第 101 行还不存在。

因此模型只能依据：

用户要求 + 已读取的代码 + 已经生成的前 99 行

来决定下一步，不能直接依据尚未生成的后续内容。

这里的“单向”主要是指生成顺序从前向后展开，不代表模型完全不能回头修改。Agent 可以调用工具重新编辑文件，但那是外层系统让模型发起新一轮生成，而不是一次生成过程天然地同时决定前后所有代码。

2. 为什么这种方式不擅长多目标权衡

关键并不是“单向自回归一定不能处理多目标”，而是：

它天然优化的是“下一个 token 看起来是否合理”，而不是显式地维护一组候选方案，并对每个方案的全部工程指标做持续、精确的比较。

假设修复一个并发 Bug，需要同时满足：

正确性
性能
内存
兼容性
可维护性

模型生成代码时，每一步实际更接近：

基于当前上下文，下一段最合理的代码是什么？

而不是：

我同时维护 20 个完整修改方案。

方案 A：
正确性 95
性能 60
内存 90
兼容性 80

方案 B：
正确性 90
性能 92
内存 70
兼容性 95

方案 C：
正确性 80
性能 97
内存 95
兼容性 60

现在比较它们的帕累托关系，再继续搜索新方案。

后者更接近 NSGA-II 等多目标优化算法的工作方式。

3. 一个直观例子

任务是：

修复缓存的并发问题，但不能明显降低 QPS，也不能增加太多内存。

LLM 可能先形成一个局部思路：

加一把全局锁可以解决并发问题。

于是开始生成：

synchronized (globalLock) {
    // cache operations
}

从“下一段代码是否合理”看，这很合理：

它直接解决竞态条件；

代码简单；

是训练数据中常见的修复模式。

但生成到后面才可能发现：

全局锁使所有请求串行化，QPS 下降严重。

问题在于，前面的实现已经把后面的生成路径限制住了。

这叫做一种生成路径依赖：

早期选择全局锁
        ↓
后续代码围绕全局锁展开
        ↓
重构成本越来越高
        ↓
模型更倾向于继续补丁，而不是推翻重来

这和人在写长代码时“已经写了很多，不愿重做”有些相似，但模型的原因不是心理惯性，而是概率生成过程和上下文结构造成的。

4. 它具体有哪些不足

第一，目标通常只以自然语言存在

LLM 一般没有内部显式变量：

correctness_score = 0.92
performance_score = 0.76
memory_score = 0.88
compatibility_score = 0.95

“性能不能下降”“保持兼容”等要求只是上下文中的文字。

模型可以理解这些文字，但不等于它会在每一步都精确更新每个目标的状态。

目标越多、上下文越长，某些目标越容易被弱化或遗漏。

第二，每一步是局部决策

下一个 token 的选择是局部的。

例如下面每一行单独看都可能合理：

const lock = new Mutex();
await lock.acquire();
updateCache();
lock.release();

但系统级结果可能不合理：

锁粒度过大
→ 热点竞争
→ 延迟上升
→ QPS 下降
→ 超时重试增加
→ 内存反而增加

局部代码合理，不代表全局系统行为合理。

第三，早期选择会约束后续路径

代码不是独立 token 的集合，而是不断累积的结构。

第 20 行选了某种架构，到了第 100 行，模型通常需要保持：

变量名一致；

API 一致；

数据结构一致；

控制流一致；

已生成代码语义一致。

因此，越往后越难彻底换方案。

这类似于在岔路口走错后，继续往前走得越远，返回成本越高。

第四，它通常只展开一个主要候选方案

一次普通生成通常是一条轨迹：

方案 A → 继续方案 A → 完成方案 A

而多目标优化更需要：

方案 A
方案 B
方案 C
方案 D
      ↓
分别评估
      ↓
淘汰劣势方案
      ↓
组合、变异、继续搜索

LLM 可以被要求生成多个方案，但如果没有测试、评估器和搜索循环，它通常只是“描述多个方案”，而不是可靠地在真实代码库中维护多个候选实现。

第五，很多目标无法仅靠读代码判断

例如：

QPS 是否下降，需要压测；

内存是否超限，需要 profiling；

并发 Bug 是否消失，需要竞态测试；

15 个模块是否兼容，需要集成测试；

尾延迟是否恶化，需要运行时数据。

LLM 只能根据代码和经验预测，不能仅靠生成过程获得这些真实反馈。

所以问题不仅是“单向”，还包括缺少外部可验证的目标函数。

5. 和 NSGA-II 的区别

NSGA-II 不是边写一个解，边猜这个解好不好，而是维护一个候选解种群：

候选方案 1
候选方案 2
候选方案 3
...
候选方案 N

每个候选方案都明确计算多个目标：

$$
f_1(x), f_2(x), \dots, f_k(x)
$$

然后做：

非支配排序；

保留帕累托前沿上的优秀方案；

保留方案多样性；

通过交叉、变异产生新方案；

重复评估。

它的核心机制就是：

同时保留多个不同的妥协方案，不急于过早押注某一个方案。

普通 LLM 生成则更像：

从一个高概率起点出发，沿着一条语言上连贯的路径一直写下去。

6. 这句话需要稍微修正

原句说：

“LLM 在输出第 100 行代码的时候，很难同时在脑海里维护并评估 10 个互斥目标的权重方案。”

这是一个便于理解的比喻，但并不完全严谨。

更准确的说法是：

普通自回归 LLM 在单次生成过程中，通常沿一条序列路径逐步做局部概率决策。它没有像多目标优化算法那样，天然维护一个显式的候选方案集合、为每个方案持续计算多个目标值，并进行帕累托排序与迭代搜索。

问题不是模型“脑容量不足”，也不是“第 100 行一定忘了要求”，而是它的默认计算范式并不是多目标搜索算法。

7. Agent 为什么仍然可能把这件事做好

自回归模型只是 Agent 的一个组成部分。外部系统可以给它补上缺失能力：

LLM 提出多个方案
↓
分别修改独立分支
↓
运行单元测试、集成测试、压测和内存测试
↓
获得明确指标
↓
比较帕累托前沿
↓
让 LLM 根据结果继续修改

此时 Agent 系统就不再是“单纯一次性写代码”，而变成：

$$
\text{生成} \rightarrow \text{测试} \rightarrow \text{评分} \rightarrow \text{选择} \rightarrow \text{再生成}
$$

所以真正的结论是：

自回归并不会让 LLM 天生无法解决多目标问题，但仅靠一次从前向后的代码生成，缺少候选方案并行搜索、显式目标评估和真实执行反馈，因此很容易得到局部合理、全局失衡的方案。
